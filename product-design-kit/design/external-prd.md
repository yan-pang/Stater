# 对外 PRD 模板

> 默认保存为 `project/domains/<domain>/delivery/prd.md`。  
> 这是领域**长期 PRD**(权威源)。版本对外交付的 `release-prd.md` 由 `/deliver` 从本文件**按稳定编号摘录组装**,不手动维护版本文件。

## 文档说明

- 对应内部设计:`project/domains/<domain>/design.md`
- 对应项目级调研:`project/research/` 下相关版本文件(按需)
- 对应项目级研发技术设计:`project/tech/` 下相关版本文件(按需)
- 面向对象:

## 条目编号规范

所有功能需求条目必须有**稳定编号**。编号规则和格式决定了 `/deliver` 能否自动识别版本变更,请严格遵守。

### 编号格式

- 固定前缀 `P-`,三位数字:`P-101`、`P-102`、`P-215`
- 编号**一旦分配永不变化**;重命名、搬移、重新排序都保留原编号
- 不同领域独立编号段,避免冲突(如用户管理用 `P-1xx`,订单用 `P-2xx`)
- 废弃条目保留编号并打标,**不删除**;仅删除会让版本 diff 失去锚点

### 条目标题格式

每个需求必须用二级标题 `## P-XXX <简短标题>` 作为段落起点。这个格式是 `/deliver` 解析 diff 的锚点。

```markdown
## P-101 新增角色审批流

**状态**: 新增@v1.0
**模块**: 角色管理
**优先级**: P0

### 业务规则
...

### 页面与交互
...

### 边界与异常
...

### 验收标准
...
```

### 状态字段

每个条目标题下第一行必须是 `**状态**:` 行,内容格式:

- `新增@v1.x` — 该版本首次引入
- `修订@v1.x` — 该版本对原条目做了修改;可累加多版本,如 `新增@v1.0;修订@v1.2;修订@v1.3`
- `废弃@v1.x` — 从该版本起不再生效,但保留条目供追溯
