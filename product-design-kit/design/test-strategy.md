# 测试策略与用例规范

> 默认保存为 `project/domains/<domain>/delivery/test-strategy.md`。  
> 与 `test-cases.md` 配套使用，先写策略，再写详细用例。  
> 目标读者通常包括 QA、开发和业务协作方，重点是验证业务逻辑、规则语义和结果可追溯性。

## 当前模板适配说明

- 当前仓库是前端 demo 模板，但测试策略仍需围绕业务逻辑、规则、状态和交付质量来设计，而不是只写页面点点点。
- 测试策略默认从 `delivery/prd.md`、`research.md`、`tech/` 和相关设计文档中提炼，需求 / 规则编号以 `delivery/prd.md` 为主。
- `test-strategy.md` 负责回答“测什么、怎么分、为什么这样分是完整的”；`test-cases.md` 负责回答“具体怎么测”。

## 核心原则

1. 逻辑覆盖：测试策略必须覆盖需求文档中的关键逻辑和关键规则。
2. 覆盖可追溯：每条测试用例都应能回溯到具体需求或规则编号。
3. 场景优先：优先用业务场景表达，不把测试退化成简单的 UI 点击脚本。
4. 边界敏感：每条关键规则至少考虑正常路径和一个边界 / 异常路径。
5. 分层清晰：策略先说明“测什么、为什么这样测”，用例再说明“具体怎么测”。

## 测试目标

| 目标 | 说明 | 重点关注 |
| --- | --- | --- |
| G1 | 主链路正确性 | 核心功能是否能走通 |
| G2 | 规则语义准确性 | 条件判断、权限、状态流转是否正确 |
| G3 | 最终结果可观察 | 用户是否能看到正确结果，系统是否有明确反馈 |
| G4 | 异常与恢复可验证 | 错误、中断、重试、撤销、回退是否有预期行为 |

## 分层策略

### 1. 推荐分层维度

可按以下一种或多种方式组合：

#### 按功能模块分层

适用于多模块并行推进的需求，重点是回答“每个模块是否完整可用”。

#### 按业务规则分层

适用于规则密集型需求，重点是回答“每条规则是否真的被覆盖”。

#### 按用户角色分层

适用于权限差异明显、同一操作对不同角色结果不同的需求。

### 2. 当前模板的观察面

在当前模板中，建议把每个用例至少落到以下观察面之一：

| 层级 | 要回答的问题 | 观察面 |
| --- | --- | --- |
| 输入层 | 输入配置是否正确 | 参数、表单、筛选条件、前置数据 |
| 决策层 | 规则是否按预期生效 | 权限、校验、状态判断、条件分支 |
| 结果层 | 用户看到的结果是否正确 | 列表结果、状态变化、操作反馈、数据变化 |
| 追溯层 | 出问题后能否定位 | 错误提示、日志线索、异常可观察性 |

### 3. 分层选择理由

建议在文档中说明为什么选择这种分层。例如：

```text
本次测试采用“模块 × 规则”双维度分层：
- 横向按页面 / 模块拆分测试集合
- 纵向按关键规则编号校验覆盖
- 交叉后可以检查每个模块的每条关键规则是否被覆盖
```

## 覆盖矩阵

覆盖矩阵是这份文档的核心证据，用来回答“为什么这些 case 是全的”。

建议至少保证：

- 每条关键需求或规则编号至少有一个用例覆盖
- 每个关键功能点至少覆盖正常路径和一个异常 / 边界路径
- 如某个交叉点不适用，明确写出不适用原因，而不是留空

## 优先级策略

| 优先级 | 划分标准 |
| --- | --- |
| P0 | 核心业务流程不通、数据准确性错误、权限安全漏洞 |
| P1 | 分支流程、关键校验规则、状态流转问题 |
| P2 | 边界情况、异常恢复、兼容性或健壮性补充验证 |

> 不要求机械地固定比例，但建议在策略中解释为什么某类 case 归为 P0 / P1 / P2。

## 回归测试策略

### 建议进入回归集的内容

| 标准 | 说明 |
| --- | --- |
| 核心业务流程 | 所有 P0 用例默认进入回归集 |
| 跨模块依赖 | 涉及模块间数据传递或联动的用例 |
| 历史缺陷 | 曾经出现过 bug 的场景 |
| 数据准确性 | 涉及计算、统计、权限判定、状态切换的用例 |

### 通常不进入回归集的内容

- 纯文案、纯样式、纯视觉排列变化
- 一次性初始化配置且后续不复用的场景
- 已被更高优先级用例完全覆盖的冗余场景

## 用例编写要求

### 用例最少应包含

| 字段 | 说明 |
| --- | --- |
| 用例编号 | 如 `TC-ROLE-001` |
| 所属模块 | 对应领域中的功能模块或页面 |
| 关联规则 | 对应需求 ID 或规则 ID |
| 优先级 | P0 / P1 / P2 |
| 测试场景 | 一句话描述场景 |
| 前置条件 | 执行前需要具备的状态 |
| 操作步骤 | 用业务语言描述，不写成纯 UI 点击脚本 |
| 预期结果 | 重点写逻辑结果和最终状态 |

### 写作要求

- 先保证规则可追溯，再写步骤细节。
- 操作步骤尽量使用业务语言，例如“创建一个角色”，而不是“点击左上角按钮”。
- 预期结果优先写系统行为和数据结果，而不是只写 toast 或弹窗文案。
- 场景举例尽量使用具体数据，而不是“若干个”“某个状态”。

## 场景覆盖要求

对于每条关键规则，至少检查以下场景是否需要覆盖：

| 场景类型 | 说明 |
| --- | --- |
| 正常路径 | 满足全部条件时，操作成功 |
| 校验失败 | 不满足输入、格式、依赖或状态规则 |
| 权限不足 | 操作者无权执行当前操作 |
| 边界值 | 临界值、极限值、空值、最大最小值 |
| 依赖冲突 | 操作与其他模块或数据存在依赖关系 |
| 恢复路径 | 失败后重试、撤销、返回、重新进入 |
| 并发 / 竞态（按需） | 多人同时操作或多次提交 |

## 权限类需求专项覆盖

如果需求包含权限计算，建议显式构造权限矩阵，而不是只写“有权限 / 无权限”。

示例：

```text
权限规则：操作人员.Games ⊇ 配置数据.Games
```

| 操作人员.Games | 配置数据.Games | 关系 | 预期 |
| --- | --- | --- | --- |
| [A, B, C] | [A, B] | 包含 | 允许操作 |
| [A, B] | [A, B] | 相等 | 允许操作 |
| [A, B] | [A, B, C] | 不包含 | 拒绝操作 |
| [A] | [B] | 无交集 | 拒绝操作 |
| [] | [A] | 空集 | 拒绝操作 |
| [A, B] | [] | 空目标 | 视规则定义 |
